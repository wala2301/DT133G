The layered (n-tier) architecture pattern organises software into horizontal layers, each responsible for a distinct concern and only communicating with adjacent layers.
A typical three-tier web application has a presentation layer (HTTP request handling, serialisation), a business logic layer (domain rules, use cases), and a data access layer (database queries, external API calls). Each layer depends only on the layer below it, not on the layer above.
This separation makes each layer independently testable and replaceable. The business logic layer can be unit tested without a database by substituting the data access layer with a fake. The data access layer can be changed from SQL to a different storage mechanism without altering business logic.
Strict layering can be overly rigid for small applications, where the overhead of mapping between layers adds complexity without proportional benefit. Clean Architecture and Hexagonal Architecture (Ports and Adapters) are variations that emphasise protecting the core domain from infrastructure concerns rather than rigid layer ordering.
