Software observability is the ability to understand the internal state of a system from its external outputs, comprising three pillars: logs, metrics, and traces.
Logs are time-stamped records of discrete events. Structured logs in JSON format are easier to query and aggregate than plain text. Log levels (DEBUG, INFO, WARNING, ERROR, CRITICAL) allow filtering by severity. Logs should capture enough context (request ID, user ID, relevant parameters) to diagnose problems without exposing sensitive data.
Metrics are numeric measurements aggregated over time, such as request rate, error rate, and response latency percentiles. They are stored in time-series databases like Prometheus and visualised in dashboards like Grafana, enabling alerting when thresholds are crossed.
Distributed tracing follows a request across multiple services by propagating a trace ID through each call. Each service records spans (units of work) with timing and metadata. Tools such as Jaeger and Zipkin reconstruct the full execution path, identifying which service or operation contributed most to latency in a distributed system.
