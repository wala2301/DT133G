Refactoring is the process of restructuring existing code without changing its observable behaviour, improving internal quality while preserving functionality.
Refactoring is performed in small, safe steps, each followed by running the test suite to confirm nothing broke. Common techniques include renaming for clarity, extracting a method from a long function, moving a method to the class it logically belongs to, replacing conditionals with polymorphism, and replacing magic numbers with named constants.
The prerequisite for safe refactoring is adequate test coverage. Without tests, it is difficult to verify that a structural change did not introduce a regression. Adding tests to legacy code is therefore often the first step before attempting significant refactoring.
Refactoring should not change behaviour; it should not mix in new features or bug fixes. Combining refactoring with feature work in the same commit obscures both the purpose of the change and the root cause of any regression it introduces. Version control commits that focus on one type of change are easier to review and bisect.
