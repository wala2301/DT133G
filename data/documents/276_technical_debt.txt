Technical debt refers to the implied cost of future rework caused by choosing a quick solution now instead of a better approach that would take longer.
Like financial debt, technical debt accrues interest: the longer it remains, the harder the surrounding code becomes to change because new features must work around the existing shortcut. Examples include hardcoded values instead of configuration, copy-pasted code instead of shared functions, and inadequate error handling added with the intent to improve later.
Technical debt is not always bad. Intentional debt—taking a shortcut to meet a deadline with a plan to refactor—is a legitimate trade-off. Unintentional debt arises from lack of skill or knowledge and is more insidious because it may not be recognised until much later.
Managing technical debt requires keeping a backlog of known issues, estimating their impact, and scheduling refactoring work alongside feature development. Code metrics such as cyclomatic complexity, duplication percentage, and test coverage help quantify and track debt over time.
