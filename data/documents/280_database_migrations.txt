Database migrations are versioned scripts that apply incremental changes to a database schema, keeping the database structure in sync with application code changes.
Each migration is a pair of operations: an upgrade that applies the change (creating a table, adding a column, changing a constraint) and a downgrade that reverses it. Migrations are stored in version control alongside the application code so schema changes are traceable and reviewable.
Migration tools such as Alembic for SQLAlchemy, Flyway, and Liquibase track which migrations have been applied using a version table in the database. When the application starts, the tool compares the applied migrations against the available scripts and runs any that are pending.
Zero-downtime migrations require special care: adding a nullable column is safe but renaming a column requires a multi-step process (add new column, backfill data, update application code to use both, remove old column) to avoid breaking running instances. Large table migrations should be batched to avoid long lock periods that block other queries.
