Git workflows define conventions for how branches are created, named, and merged in a team, providing a shared structure for collaborative development.
GitFlow is a branching model with long-lived main and develop branches. Feature branches are created from develop, merged back when complete, and release branches are created from develop to prepare a release before merging into both main and develop. Hotfix branches are created directly from main for urgent production fixes.
Trunk-based development keeps all developers working on a single main branch, using short-lived feature branches or direct commits. Feature flags hide incomplete work in production. This model reduces merge conflicts and supports continuous integration by keeping branches small and short-lived.
GitHub Flow is a simplified model where the main branch is always deployable. Developers branch from main, open a pull request, and merge after review. It suits teams that deploy frequently and prefer simplicity over the structure of GitFlow. The choice of workflow should match the team's deployment cadence and release process.
