Clean code refers to code that is easy to read, understand, and modify, reducing the long-term cost of maintaining a software system.
Meaningful names are fundamental: variables, functions, and classes should reveal their intent. Names like process_data or d are unhelpful; names like calculate_monthly_revenue or elapsed_days communicate purpose without requiring a comment.
Functions should do one thing and do it well. A function that is too long likely performs several distinct tasks and should be decomposed. Functions with many parameters often indicate that a parameter object or class is more appropriate. Side effects should be minimised and documented.
Code duplication is a maintenance liability because a bug or requirement change must be fixed in multiple places. The DRY (Don't Repeat Yourself) principle advocates extracting repeated logic into a shared abstraction. However, premature abstraction of superficially similar code can introduce coupling between unrelated concepts, so duplication should be eliminated when the commonality is real, not coincidental.
